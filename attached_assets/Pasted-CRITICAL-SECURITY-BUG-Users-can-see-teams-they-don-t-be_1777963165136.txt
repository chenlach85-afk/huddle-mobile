The bug

A coach should see ONLY teams where they are a member (created the team, or were added as a coach/assistant_coach)
A player should see ONLY teams where they are a member as a player or parent

Step 3 — Fix the RLS policies
Replace the existing teams and team_members policies with these strict, correct versions:
sql-- Make sure RLS is enabled
alter table public.teams enable row level security;
alter table public.team_members enable row level security;

-- ===========================================
-- TEAMS TABLE POLICIES
-- ===========================================

-- Drop ALL existing policies on teams (we're rewriting them)
do $$
declare pol record;
begin
  for pol in (select policyname from pg_policies where tablename = 'teams' and schemaname = 'public')
  loop
    execute format('drop policy if exists %I on public.teams', pol.policyname);
  end loop;
end $$;

-- Helper: is the current user a member of this team (any role)?
create or replace function public.is_team_member(team_uuid uuid)
returns boolean language sql security definer stable
set search_path = public as $$
  select exists(
    select 1 from public.team_members
    where team_id = team_uuid
      and user_id = auth.uid()
      and status = 'active'
  );
$$;

-- Helper: is the current user a coach of this team?
create or replace function public.is_team_coach(team_uuid uuid)
returns boolean language sql security definer stable
set search_path = public as $$
  select exists(
    select 1 from public.team_members
    where team_id = team_uuid
      and user_id = auth.uid()
      and role in ('coach', 'assistant_coach')
      and status = 'active'
  );
$$;

-- Helper: is the current user the team owner (created the team)?
create or replace function public.is_team_owner(team_uuid uuid)
returns boolean language sql security definer stable
set search_path = public as $$
  select exists(
    select 1 from public.teams
    where id = team_uuid and created_by = auth.uid()
  );
$$;

-- SELECT: only members of a team can see it
create policy "members view team" on public.teams
  for select to authenticated
  using (public.is_team_member(id));

-- INSERT: any authenticated user can create a team (becomes the owner via created_by)
create policy "users create teams" on public.teams
  for insert to authenticated
  with check (created_by = auth.uid());

-- UPDATE: only the owner can update team metadata
create policy "owner updates team" on public.teams
  for update to authenticated
  using (public.is_team_owner(id))
  with check (public.is_team_owner(id));

-- DELETE: only the owner can delete the team
create policy "owner deletes team" on public.teams
  for delete to authenticated
  using (public.is_team_owner(id));

-- ===========================================
-- TEAM_MEMBERS TABLE POLICIES
-- ===========================================

-- Drop ALL existing policies on team_members
do $$
declare pol record;
begin
  for pol in (select policyname from pg_policies where tablename = 'team_members' and schemaname = 'public')
  loop
    execute format('drop policy if exists %I on public.team_members', pol.policyname);
  end loop;
end $$;

-- SELECT: members can see other members of teams they're in
create policy "members view roster" on public.team_members
  for select to authenticated
  using (public.is_team_member(team_id));

-- SELECT (special case): users can always see their own membership rows even if they're pending
create policy "users see own memberships" on public.team_members
  for select to authenticated

-- INSERT: users can only add themselves (when joining via invite code)
create policy "users join teams" on public.team_members
  for insert to authenticated

-- INSERT: coaches can add players to their teams
create policy "coaches add players" on public.team_members
  for insert to authenticated
  with check (
    public.is_team_coach(team_id)
    and role in ('player', 'parent')
  );

-- INSERT: owner can add coaches to their team
create policy "owner adds coaches" on public.team_members

-- UPDATE: coaches can update player rows (jersey, position, etc.)

Step 8 — Add a logged warning for service role usage
To prevent this from happening again, add runtime detection. In your service role client wrapper:
typescript// lib/supabase-admin.ts
import { createClient } from '@supabase/supabase-js';

let warned = false;

export function getSupabaseAdmin() {
  if (!warned && process.env.NODE_ENV !== 'production') {
    const stack = new Error().stack;
    // Whitelist known-safe call sites
    const safe = [
      '/api/admin/',
      '/api/webhooks/',
      '/api/auth/accept-invite',
      'lib/admin-',
      'cron/',
    ];
    if (!safe.some(p => stack?.includes(p))) {
      console.warn('⚠️ SERVICE ROLE used outside admin/webhook context — RLS bypassed!\n', stack);
    }
  }
  return createClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.SUPABASE_SERVICE_ROLE_KEY!
  );
}

That's the prompt. Some honest notes:
On the severity — this is the biggest bug you've had. A user seeing other users' data isn't a UX glitch, it's a privacy violation. If you eventually charge customers, this is the kind of thing that gets you sued or breaches GDPR. Fix it before doing anything else.
On RLS as the only real defense — UI-level filtering ("just hide the other teams in the React component") is not a fix. An attacker can open DevTools, find the API call, and hit it directly with their own user token. Only the database RLS policy actually prevents the data from being returned. The prompt above fixes RLS first, which is correct.
On the service role gotcha — this is the #1 cause of RLS data leaks in Supabase apps. The pattern is: agent builds a feature, hits a permission error, "fixes" it by switching to service role, ships it. Now the feature works AND is leaking data. The fix is to remove service role from non-admin paths and ensure RLS policies actually allow legitimate access.
