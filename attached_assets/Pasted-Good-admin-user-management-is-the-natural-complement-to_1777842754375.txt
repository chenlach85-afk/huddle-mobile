Good — admin user management is the natural complement to invitations. Let me give you a focused prompt that adds full account control without breaking what you already have.

A few important things to think about before I write it, because "delete a user" sounds simple but has nasty edge cases that bite you in production:

What "delete" actually means — pick your approach
There are three different actions, and you probably want all of them but for different situations:

Action	What it does	When to use
Suspend	User can't log in; data stays	Temporary block, payment issues, behavior warning
Soft delete	User marked deleted, data anonymized but kept; user can't log in or recover	Standard "delete" — preserves history of what they did
Hard delete	User row + all their data permanently destroyed	GDPR/privacy requests, mistakes, test accounts
Why this matters: if a coach who managed 5 teams gets hard-deleted, what happens to those teams? Their messages? Their events? The task assignments they created? If you just DELETE FROM auth.users and let CASCADE handle it, you nuke real data that other users (the players) might still need.

The professional pattern is: soft delete by default, hard delete as a separate destructive action. I'll build both.

What about the data they own?
When you delete a coach, you have three options for their teams:

Cascade delete — team and all its data disappears (clean but destructive)
Transfer ownership — admin picks a new owner before deletion
Archive — team stays, marked as archived, players keep historical access read-only
Best practice: make this a choice in the UI. Don't bake one option into the code. The admin sees a confirmation dialog with the three options every time.

-- Add deletion tracking fields to profiles
alter table public.profiles
  add column if not exists deleted_at timestamptz,
  add column if not exists deleted_by uuid references auth.users(id),
  add column if not exists deletion_reason text,
  add column if not exists suspended_at timestamptz,
  add column if not exists suspended_by uuid references auth.users(id),
  add column if not exists suspension_reason text;

-- Audit log for admin actions
create table if not exists public.admin_audit_log (
  id uuid primary key default uuid_generate_v4(),
  admin_id uuid not null references auth.users(id),
  action text not null check (action in (
    'user_suspended', 'user_reactivated', 'user_soft_deleted', 'user_hard_deleted',
    'user_promoted_admin', 'user_demoted_admin', 'user_password_reset',
    'team_archived', 'team_transferred', 'team_deleted',
    'invitation_sent', 'invitation_revoked'
  )),
  target_user_id uuid references auth.users(id),
  target_team_id uuid,
  metadata jsonb,
  created_at timestamptz default now()
);

create index admin_audit_log_admin_idx on admin_audit_log(admin_id, created_at desc);
create index admin_audit_log_target_idx on admin_audit_log(target_user_id);

-- Add archived flag to teams
alter table public.teams
  add column if not exists archived_at timestamptz,
  add column if not exists archived_by uuid references auth.users(id),
  add column if not exists archived_reason text;

-- RLS for audit log: only admins can read; nobody can update/delete (immutable)
alter table public.admin_audit_log enable row level security;
create policy "admins read audit log" on public.admin_audit_log for select
  using (exists(select 1 from profiles where id = auth.uid() and is_platform_admin = true));
-- Inserts happen via service role only (from API routes), no policy needed for that

-- Helpful view: active users (not deleted, not suspended)
create or replace view public.active_users as
  select p.*, u.email as auth_email, u.created_at as auth_created_at, u.last_sign_in_at
  from profiles p
  join auth.users u on u.id = p.id
  where p.deleted_at is null and p.account_status != 'suspended';

async function requireAdmin(request: Request) {
  const supabase = createServerClient(/* ... */);

POST /api/admin/users/:id/soft-delete
The standard "delete" action. Marks the user as deleted but keeps their row.

json
// Body
{
  "reason": "string?",
  "team_action": "archive" | "transfer" | "delete",
  "transfer_to_user_id": "uuid?"  // required if team_action === 'transfer'
}
Behavior:

Validate team_action. If transfer, ensure transfer_to_user_id exists and isn't deleted.
For each team this user owns (teams.created_by = userId):
If team_action === 'archive': set archived_at = now(), archived_by = admin.id, archived_reason = 'Owner soft-deleted'. Players in archived teams get read-only access.
If team_action === 'transfer': update teams.created_by = transfer_to_user_id. Update team_members to make new user a coach role.
If team_action === 'delete': cascade delete the team (events, messages, etc. all gone via existing FK cascade).
Anonymize the profile: full_name = 'Deleted User', email = 'deleted-{user_id}@deleted.local', phone = null, avatar_url = null. Keep id intact so foreign keys don't break.
Set deleted_at = now(), deleted_by = admin.id, deletion_reason = body.reason, account_status = 'suspended'.
Disable the auth user: supabase.auth.admin.updateUserById(userId, { ban_duration: '876000h' }) (100 years).
Sign out all sessions.
Log to audit with metadata containing team_action and counts.
Don't send a confirmation email (the user is being removed, not notified).
Soft-deleted users do NOT count toward platform stats but their historical records (messages they sent, events they created) remain visible to other users with the anonymized name.

POST /api/admin/users/:id/hard-delete
DESTRUCTIVE. Use rarely. Requires extra confirmation in the UI.

json
// Body
{
  "reason": "string?",
  "team_action": "delete" | "transfer",
  "transfer_to_user_id": "uuid?",
  "confirmation_text": "string"  // must match exact phrase like "DELETE PERMANENTLY"
}
Behavior:

Verify confirmation_text === 'DELETE PERMANENTLY'. Return 400 otherwise.
Handle teams per team_action (only delete or transfer allowed for hard delete — no archive option since there'll be no owner to revive it).
Log audit BEFORE deletion (user_hard_deleted with snapshot of user data in metadata so audit trail isn't lost).
Delete the auth user via service role: supabase.auth.admin.deleteUser(userId). This cascades through all FK relationships in your schema (positions, alerts, messages they sent, etc.).
Don't try to anonymize first — the deletion is the anonymization.
Safety rails:

Can't hard-delete yourself
Can't hard-delete the last platform admin
Can't hard-delete if user has active subscription (must cancel first via Stripe)

Sign-in flow change
Update the sign-in handler to check account status AFTER successful auth:

typescript
// In /api/auth/signin or wherever login happens
const { data, error } = await supabase.auth.signInWithPassword({ email, password });
if (error) return /* normal error */;

const { data: profile } = await supabase.from('profiles').select('account_status, deleted_at').eq('id', data.user.id).single();

if (profile.deleted_at) {
  await supabase.auth.signOut();
  return error('Account no longer exists', 'account_deleted');
}
if (profile.account_status === 'suspended') {
  await supabase.auth.signOut();
  return error('Account is suspended. Please contact support.', 'account_suspended');
}
// proceed with login

That's the prompt. Some honest notes:

On testing the destructive stuff — create 2-3 test accounts via the invitation flow first, then practice suspend → reactivate → soft delete → hard delete on those. Don't experiment on real coach accounts. Once you've proven it works on test data, you'll trust it for production.

On audit logs in practice — you'll be tempted to skip them ("it's just me as admin"). Don't. The first time you accidentally delete the wrong user (everyone does this once), the audit log is what saves you. It's also what you show a regulator if you ever face a privacy complaint.

On the "active subscription" check before hard delete — I added this rule but only conceptually. If you haven't built billing yet, the agent might skip it or implement it as a no-op stub. That's fine. When you add Stripe later (the billing addendum prompt I gave you earlier), wire it up properly then.

On data preservation tradeoffs — soft delete is almost always right. Hard delete should be reserved for: GDPR/privacy requests, accidental signups, spam accounts, and your own test accounts. If a real coach asks "delete my account," soft delete is enough — their data is anonymized and they can't log in. Only escalate to hard delete if they specifically ask for full data removal.

One feature I deliberately left out: bulk operations (suspend 50 users at once). Easy to add later if you ever need it. For early-stage admin work, one-at-a-time is safer.
