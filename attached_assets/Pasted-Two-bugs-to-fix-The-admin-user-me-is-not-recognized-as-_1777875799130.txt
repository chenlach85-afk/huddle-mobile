Common bugs to specifically check

Some implementations cache the profile in a React context loaded once at app boot. If the admin flag was flipped after I logged in, the cached context still says false. Force a profile refetch on mount, or clear the cache when the profile is updated.

Hardening (recommended): block orphan profiles
Even with all the above, technically someone could call Supabase's auth REST API directly with the anon key and create an auth.users row. They wouldn't have a profile so they couldn't really use the app, but it's untidy.
Add this trigger to make it bulletproof:
sql-- Replace the existing handle_new_user trigger
create or replace function public.handle_new_user()
returns trigger
language plpgsql
security definer set search_path = public
as $$
declare
  invitation_exists boolean;
begin
  -- Check if there's a pending invitation for this email
  select exists(
    select 1 from platform_invitations
    where email = new.email
      and status = 'pending'
      and expires_at > now()
  ) into invitation_exists;

  -- Allow if invitation exists OR if this is the first user (initial admin)
  if not invitation_exists and (select count(*) from profiles) > 0 then
    raise exception 'Signup requires a valid invitation';
  end if;

  insert into public.profiles (id, email, full_name)
  values (new.id, new.email, coalesce(new.raw_user_meta_data->>'full_name', ''));
  return new;
end;
$$;

drop trigger if exists on_auth_user_created on auth.users;
create trigger on_auth_user_created
  after insert on auth.users
  for each row execute procedure public.handle_new_user();
Now even direct Supabase API attacks will fail at the database level — the trigger raises an exception and the user creation is rolled back.
Note: this means /api/auth/accept-invite must keep the invitation in pending status until AFTER supabase.auth.admin.createUser succeeds, then mark it accepted. Don't update the invitation first.

That's the prompt. Some practical notes:
On finding your admin row — when the agent runs the diagnostic SQL, look at the result. The first user created (order by created_at asc limit 1) is almost certainly you. If you signed up multiple times during testing, look for the row with the email you actually use. Tell the agent which email is yours when it asks.
On the "first user is admin" rule — the trigger I added has a clever escape hatch: (select count(*) from profiles) > 0. This means the very first signup ever (when profiles is empty) bypasses the invitation requirement. After that, every signup requires an invitation. That's how you'll bootstrap if you ever wipe the database and start over.
On the test that uses incognito — actually do this. Open Chrome incognito, visit your app's signup URL, try to register. If you see a form, the gate isn't working. If you see the "invitation only" message, you're good. This is the single most important test.
On Supabase Auth settings — go to your Supabase dashboard → Authentication → Email Templates and customize the password recovery email if you haven't. The default English template won't make sense for Hebrew users. You can leave the auth signup template alone since you're not using it.
If the agent gets stuck — the most common failure is admin cookies/sessions caching old profile data. If after the fix you still see "not authorized," do a hard browser refresh, clear cookies for the app domain, and log in fresh. That solves about 70% of "I should be admin but I'm not" issues.
