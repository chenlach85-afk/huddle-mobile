Bug 4 — Coaches need to invite players via email or link
Coaches currently have no clean way to add players to their teams. Build two parallel flows: invite by email (with magic link, like the platform invitations) and invite by shareable link (the existing team invite code, made more usable).
Conceptual model
Coaches and only coaches can invite players to their teams. There are two flows:
Flow A — Email invitation (one specific person): Coach types an email address, system sends an invitation email to that exact person. Recipient clicks the link, lands on a pre-filled signup page where their email is locked.
Flow B — Shareable link (any person with the link): Coach copies a link or shares it via WhatsApp. Anyone with the link can use it to join the team — useful for parent groups, WhatsApp blasts, etc.
Both flows result in a team_members row being created with role = 'player' (or 'parent' if the coach selects).
Database
We already have teams.invite_code for Flow B. Add a separate table for Flow A:
sqlcreate table if not exists public.team_invitations (
  id uuid primary key default uuid_generate_v4(),
  team_id uuid not null references teams(id) on delete cascade,
  email text not null,
  invited_role text not null default 'player' check (invited_role in ('player', 'parent', 'assistant_coach')),
  -- Prefilled fields, optional, the coach can pre-set jersey number etc
  prefill_jersey_number int,
  prefill_position text,
  prefill_full_name text,
  -- Personal message from coach, shown in email and on accept page
  personal_message text,
  token text unique not null default replace(gen_random_uuid()::text || gen_random_uuid()::text, '-', ''),
  status text default 'pending' check (status in ('pending', 'accepted', 'revoked', 'expired')),
  accepted_by_user_id uuid references auth.users(id),
  accepted_at timestamptz,
  expires_at timestamptz default now() + interval '14 days',
  invited_by uuid references auth.users(id),
  email_sent_at timestamptz,
  email_send_count int default 0,
  created_at timestamptz default now()
);

create index if not exists team_invitations_team_idx on team_invitations(team_id);
create index if not exists team_invitations_email_idx on team_invitations(email);
create index if not exists team_invitations_token_idx on team_invitations(token);
create index if not exists team_invitations_status_idx on team_invitations(status);

-- Allow anonymous read for token lookup (the public landing page needs this)
alter table public.team_invitations enable row level security;

create policy "anyone can read team invitation by token"
  on public.team_invitations for select to anon, authenticated
  using (status = 'pending' and expires_at > now());

-- Coaches manage invitations for their own teams
create policy "coaches manage team invitations"
  on public.team_invitations for all to authenticated
  using (is_team_coach(team_id))
  with check (is_team_coach(team_id));
Update handle_new_user trigger
The existing trigger blocks signups without a platform invitation OR a joining_team_id in metadata. Extend it to also accept a team_invitation token:
sqlcreate or replace function public.handle_new_user()
returns trigger
language plpgsql
security definer set search_path = public
as $$
declare
  has_platform_invite boolean;
  has_team_code text;
  has_team_invitation boolean;
begin
  select exists(
    select 1 from platform_invitations
    where email = new.email and status = 'pending' and expires_at > now()
  ) into has_platform_invite;

  -- joining_team_id is set when user joins via /join/[code] (shareable link, Flow B)
  has_team_code := new.raw_user_meta_data->>'joining_team_id';

  -- joining_team_invitation_id is set when user accepts /team-invite/[token] (Flow A)
  select exists(
    select 1 from team_invitations
    where id::text = new.raw_user_meta_data->>'joining_team_invitation_id'
      and status = 'pending'
      and expires_at > now()
      and email = new.email
  ) into has_team_invitation;

  if not has_platform_invite
     and has_team_code is null
     and not has_team_invitation
     and (select count(*) from profiles) > 0 then
    raise exception 'Signup requires either a platform invitation, team invite code, or team invitation';
  end if;

  insert into public.profiles (id, email, full_name, account_status)
  values (
    new.id,
    new.email,
    coalesce(new.raw_user_meta_data->>'full_name', ''),
    'active'
  );
  return new;
end;
$$;

That's the full combined prompt. A few practical notes:
On bug 4's two flows being parallel — coaches will use both, often. The email invitation is for "one specific player I want to add" (most common). The shareable link is for "I'm starting a new season, share this with all the parents at once." Don't pick one, build both.
On the team invite URL difference — /join/[code] is for shareable links (the existing flow, just made more usable), /team-invite/[token] is for email invitations (one-time, locked to a specific email). Two different URL patterns, two different tables, two different behaviors. The agent will be tempted to merge them — push back, they're not the same thing.
On the "already logged in" flow for recipients — this matters more than you'd think. If a coach invites a player who already has a HUDDLE account (because they're already a member of another team), clicking the email link should NOT create a duplicate account. The flow should detect this and just add them to the new team. The prompt covers this in the test cases.
On the email branding being team-specific — this is a small detail that makes a big difference. When a coach invites a parent and the email shows the team's actual logo and color (not generic HUDDLE orange), parents trust it more. They feel "this is my kid's team" not "this is a generic SaaS." Worth the extra 30 minutes to implement properly.
